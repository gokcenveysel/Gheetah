What is Gheetah?

Gheetah is a powerful Test Orchestration Platform designed to streamline the execution and management of BDD (Behavior-Driven Development) test automation projects. Built as a cross-platform solution with a .NET 8.0 MVC backend, it features a modern and intuitive frontend designed using Tabler.io.

Key Features & Capabilities:
  • Seamless Test Execution:
    • Supports BDD test projects in C# (Reqnroll/SpecFlow) or Java (TestNG-Cucumber/JUnit-Cucumber).
    • Run tests remotely via Gheetah Agent or directly on the Gheetah Server.
    • Target specific scenarios using TAG filtering.
    • View real-time test results with structured reporting.
  • Enterprise-Ready Platform:
    • Customizable Dashboard for analytics and visibility.
    • User & Role Management for secure access control.
    • Integrations with Azure DevOps, Jenkins, GitLab, Email, and more.
    • Scalable for enterprise environments.
  • Lightweight & Easy to Use:
    • No database dependency‚Äîstores data as JSON files in the /Data directory.

System Requirements

Gheetah Server

Core Platform

The central orchestration hub that manages test execution, reporting, and integrations. Designed for enterprise-scale deployment.

Infrastructure

  • Operating System: Windows Server 2016/2019/2022 or Linux (RHEL 8+, Ubuntu 20.04+)
  • Compute: 4 CPU cores (8+ recommended for concurrent executions)
  • Memory: 8GB RAM minimum (16GB recommended for production)
  • Storage: 5GB available space (SSD strongly recommended)

Software Dependencies

.NET 8.0 ASP.NET Core

Self-contained deployments available for environments without runtime pre-installation

For Java Project Support:
  • JDK: OpenJDK 11+ or Oracle JDK 11+
  • Build Tools: Maven 3.6+ or Gradle 7.4+
  • Environment: Properly configured JAVA_HOME and build tool paths

Execution Agent

Test Runner

Lightweight components that execute test scenarios in distributed environments with minimal overhead.

Infrastructure

  • Operating System: Windows 10/11 or Linux (x64 distributions)
  • Compute: 2 CPU cores minimum (4+ recommended for parallel execution)
  • Memory: 4GB RAM minimum (scales with test suite complexity)

Runtime Environments

.NET 8.0 Java 11+
Language-Specific Requirements:
C#/SpecFlow
  • NuGet package restore capability
  • Build tools: MSBuild or dotnet CLI
Java/Cucumber
  • Maven or Gradle installation
  • Network access to artifact repositories
  • Proper JAVA_HOME configuration

Deployment Notes

For optimal performance in containerized environments:

  • Ensure sufficient CPU shares (minimum 2 vCPUs per agent container)
  • Mount persistent storage for test artifacts and logs
  • Configure network access to version control systems and package repositories

For detailed sizing guidance based on your test suite characteristics, contact our solutions team.

First Installation Wizard

When you deploy Gheetah to a compatible server meeting the system requirements or run it via Visual Studio, you will be greeted by the "First Installation" wizard. This intuitive setup process guides you through the initial configuration steps to ensure a smooth onboarding experience.

Configuration Steps

  1. SSO Provider Selection
    • Azure AD: If you select Azure AD, you will be prompted to configure the Azure AD integration settings including tenant ID, application ID, and secret key.
    • Google Workspace: If you choose Google Workspace, you will need to set up the OAuth 2.0 credentials and configure domain-wide delegation.
  2. Manage Groups: Define and organize user groups to streamline access control. The system supports hierarchical group structures with inheritance capabilities.
  3. Add Permissions to Groups: Assign specific permissions to the defined groups to tailor access levels. Granular permissions are available for all system modules.
Important Note

Default groups and roles are pre-configured with industry best practices for optimal security and usability. We strongly recommend retaining these settings unless specific customization is required. If modifications are made, you may need to update the corresponding [Auth] attributes in the codebase to maintain consistency between configuration and implementation.

Upon completing these steps, click "Save Configuration" to finalize the initial setup. The application will automatically:

  • Validate all configuration parameters
  • Initialize the necessary system resources
  • Create the default data structures
  • Redirect you to the related(Azure, Google or Custom) Login screen

Google Workspace Single Sign-On (SSO) Integration

This guide explains how to configure Google Workspace as an identity provider (IdP) for secure Single Sign-On (SSO) authentication.

Google Cloud Console ‚Üí OAuth Consent Screen ‚Üí Credentials

  1. Access Google Cloud Console
    • Go to the Google Cloud Console.
    • Select an existing project or create a new one.
    • Ensure Google Workspace APIs are enabled under APIs & Services > Library.
  2. Configure OAuth Consent Screen
    • Navigate to APIs & Services > OAuth Consent Screen.
    • Select External (for all users) or Internal (for Workspace-only).
    • Fill in:
      • Application Name (e.g., "MyApp SSO")
      • Support Email (admin contact)
      • Authorized Domains (your company domain)
    • Add required scopes: email, profile, openid.
  3. Create OAuth 2.0 Client ID
    • Go to APIs & Services > Credentials.
    • Click Create Credentials > OAuth Client ID.
    • Select Web Application.
    • Under Authorized Redirect URIs, add:
      ‚ö†Ô∏è Required Callback URL:
      https://yourdomain.com/sso/google/callback
      This must match exactly; otherwise, authentication will fail.
  4. Save Credentials & Configure Your App
    • Copy the Client ID and Client Secret.
    • In your application's SSO settings, enter:
      • Google Client ID
      • Google Client Secret
      • Callback URL: /sso/google/callback

Microsoft Azure Active Directory (AD) SSO Integration

Follow these steps to set up Azure AD as an identity provider (IdP) for secure Single Sign-On (SSO).

Azure Portal ‚Üí App Registrations ‚Üí Authentication ‚Üí API Permissions

  1. Access Azure Active Directory
    • Log in to the Microsoft Azure Portal.
    • Go to Azure Active Directory > App Registrations.
    • Click New Registration.
  2. Register a New Application
    • Enter a Name (e.g., "MyApp SSO").
    • Select Accounts in this organizational directory only (for single-tenant).
    • Under Redirect URI, select Web and enter:
      ‚ö†Ô∏è Required Callback URL:
      https://yourdomain.com/sso/azure/callback
      This must be exactly /sso/azure/callback for authentication to work.
    • Click Register.
  3. Configure Authentication Settings
    • Go to Authentication.
    • Ensure ID Tokens is checked.
    • Add additional redirect URIs if needed (e.g., for dev/staging environments).
  4. Set API Permissions
    • Navigate to API Permissions.
    • Click Add a permission > Microsoft Graph.
    • Add delegated permissions:
      • openid
      • profile
      • email
    • Click Grant admin consent.
  5. Generate Client Secret
    • Go to Certificates & Secrets.
    • Click New client secret.
    • Set an expiration (recommended: 12 months).
    • Copy the Secret Value (you won‚Äôt see it again).
  6. Configure Your Application
    • In your app‚Äôs SSO settings, enter:
      • Domain
      • Azure Tenant ID (found in Azure AD > Overview)
      • Application (Client) ID
      • Client Secret
      • Callback URL: /sso/azure/callback

  • Error: "AADSTS50011 - Redirect URI mismatch" ‚Üí Verify the callback URL is registered in Azure AD.

Local Authentication (No SSO)

Gheetah's built-in authentication system provides a secure local user management solution without requiring external identity providers. This is the default option during initial setup.

üöÄ Zero Configuration Required
No additional setup is needed. Simply select "No SSO" during initial configuration to activate Gheetah's native authentication system.
Key Features:
  • Automatic JWT-Based Authentication
    Gheetah automatically implements OpenID-compatible authentication using JWT Bearer tokens for secure API access.
  • Seamless User Onboarding
    Custom registration and login endpoints are generated automatically when you choose this option.
  • Complete User Management
    • User self-registration (configurable)
    • Secure password reset functionality
    • Account locking after failed attempts
    • Email verification workflows
Implementation Details:

When you select "No SSO" during initial setup:

  1. Gheetah generates:
    • account/register endpoint (POST)
    • account/login endpoint (POST)
    • JWT token generation/validation system
  2. All authentication flows use:
    • OpenID-compatible JWT tokens
    • HS256 signing algorithm (configurable)
    • Secure HTTP-only cookies for web sessions

Users created through this system are immediately available in Gheetah with default basic user permissions (configurable in admin panel).

Defined Groups

Gheetah features a fully customizable group management system where you can define groups with any naming convention that suits your organization's structure.

Flexible Group Management

While you can create completely custom groups tailored to your company or team structure, Gheetah comes with pre-configured default groups defined in the appsettings.json configuration file for immediate usability.

Default Group Configuration

Group ID Group Name Description Access Level
admin-grp Admin Full system administration privileges Full Access
lead-grp Lead Project management capabilities Elevated Access
runner-grp Runner Execution privileges for assigned test runs Limited Access

Key Features of Group Management

  • Custom Naming: Groups are managed by unique IDs, allowing complete flexibility in naming conventions
  • Initial Setup Configuration: Permissions can be assigned to groups during the first installation wizard
  • Enterprise Ready: Supports complex organizational structures with hierarchical permissions
  • Dynamic Updates: Groups can be modified post-installation through the admin interface
Implementation Note

The group IDs (admin-grp, lead-grp, runner-grp) are hardcoded references in the permission system. While you can change the display names, we recommend maintaining the original IDs for compatibility with future updates.

Role and Permission Management

Gheetah provides a comprehensive role-based access control (RBAC) system with fully customizable permissions tailored to your organizational needs.

Customizable Role System

While you can create completely custom roles with specific naming conventions, Gheetah comes pre-configured with default roles and permissions defined in the appsettings.json file for immediate deployment.

Key Features

  • Flexible Role Assignment: Roles can be assigned to groups during initial setup or modified later
  • ID-Based Management: Roles are managed by unique IDs, allowing complete naming flexibility
  • Granular Permissions: Fine-grained control over system actions and capabilities
Important Implementation Notes
  1. Authentication Mechanism: The authorization system references role IDs directly. If you modify these IDs, you must update:
    • The Program.cs configuration
    • All Controller classes with [Auth] attributes
  2. Post-Setup Modifications: While role permissions can theoretically be modified after initial setup, this functionality remains untested in production environments.

Dashboard Management

Initial Access Privileges

Upon first login after installation, the initial user account is automatically registered with Administrator privileges, granting full access to all Gheetah functionality. This includes comprehensive dashboard management capabilities.

Dashboard Customization Features

Personalized Layouts

Click the gear icon to access dashboard customization options where you can:

  • Add, remove, and configure widgets
  • Rearrange elements via drag-and-drop
  • Save personalized layouts
Widget Management

Gheetah offers flexible widget configuration:

  • Resizable components with multiple size presets
  • Custom positioning on the dashboard grid
  • User-specific widget preferences

User-Specific Dashboard Experience

The dashboard system is user-specific, meaning:

  • Each user maintains their own saved dashboard configuration
  • Layout changes are automatically persisted per user
  • Administrators can set default templates for new users
Visual Indicator

The dashboard displays a privilege indicator showing your current access level. Administrators will see additional management controls and system-wide options not available to standard users.

Note for First-Time Users

If you're the initial setup user, ensure you:

  1. Properly configure your admin dashboard before creating additional user accounts
  2. Consider setting up template dashboards for different user roles
