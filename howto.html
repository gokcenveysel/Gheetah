Project Folder Configuration

Critical System Configuration

While this interface appears simple, it represents one of the most crucial settings in Gheetah Test Orchestration and must be properly configured for the system to function.

Directory Structure Specification

This screen defines the root directory where all BDD test automation projects will be stored when cloned or uploaded to your Gheetah instance.

Default Fallback Behavior

If no directory is specified:

  • Projects will default to Gheetah\CopiedProjects in the application's root directory
  • The system will appear to accept clone/upload operations
  • However, critical failures will occur because:
    • The projects-folder.json file will remain empty
    • Test scenarios cannot execute - neither remotely on Agents nor locally on the Gheetah server
    • Build processes will fail completely

Technical Implementation

The system relies on the projects-folder.json configuration file to:

  • Track the location of all test projects
  • Manage file operations during test execution
  • Coordinate between the Gheetah server and remote Agents
  • Maintain project isolation and organization

Without this configuration: The entire test orchestration pipeline will be non-functional, as all workflows depend on the directory path specified in projects-folder.json.

Configuration Best Practices
  • Choose a dedicated directory with sufficient storage space
  • Ensure the location has proper read/write permissions
  • Consider using high-performance storage for better execution speed
  • For enterprise deployments, use a network-accessible location
  • Document the chosen location for administrative reference

What is Gheetah?

Gheetah is a powerful Test Orchestration Platform designed to streamline the execution and management of BDD (Behavior-Driven Development) test automation projects. Built as a cross-platform solution with a .NET 8.0 MVC backend, it features a modern and intuitive frontend designed using Tabler.io.

Key Features & Capabilities:
  • Seamless Test Execution:
    • Supports BDD test projects in C# (Reqnroll/SpecFlow) or Java (TestNG-Cucumber/JUnit-Cucumber).
    • Run tests remotely via Gheetah Agent or directly on the Gheetah Server.
    • Target specific scenarios using TAG filtering.
    • View real-time test results with structured reporting.
  • Enterprise-Ready Platform:
    • Customizable Dashboard for analytics and visibility.
    • User & Role Management for secure access control.
    • Integrations with Azure DevOps, Jenkins, GitLab, Email, and more.
    • Scalable for enterprise environments.
  • Lightweight & Easy to Use:
    • No database dependency—stores data as JSON files in the /Data directory.

Google Workspace Single Sign-On (SSO) Integration

This guide explains how to configure Google Workspace as an identity provider (IdP) for secure Single Sign-On (SSO) authentication.

Google Cloud Console → OAuth Consent Screen → Credentials

  1. Access Google Cloud Console
    • Go to the Google Cloud Console.
    • Select an existing project or create a new one.
    • Ensure Google Workspace APIs are enabled under APIs & Services > Library.
  2. Configure OAuth Consent Screen
    • Navigate to APIs & Services > OAuth Consent Screen.
    • Select External (for all users) or Internal (for Workspace-only).
    • Fill in:
      • Application Name (e.g., "MyApp SSO")
      • Support Email (admin contact)
      • Authorized Domains (your company domain)
    • Add required scopes: email, profile, openid.
  3. Create OAuth 2.0 Client ID
    • Go to APIs & Services > Credentials.
    • Click Create Credentials > OAuth Client ID.
    • Select Web Application.
    • Under Authorized Redirect URIs, add:
      ⚠️ Required Callback URL:
      https://yourdomain.com/sso/google/callback
      This must match exactly; otherwise, authentication will fail.
  4. Save Credentials & Configure Your App
    • Copy the Client ID and Client Secret.
    • In your application's SSO settings, enter:
      • Google Client ID
      • Google Client Secret
      • Callback URL: /sso/google/callback
🔍 Troubleshooting
  • Error: "Redirect URI mismatch" → Verify the callback URL matches exactly.
  • Error: "Invalid scope" → Ensure openid, email, and profile are included.
  • 403 Access Denied → Check domain restrictions in Google Workspace Admin.

Microsoft Azure Active Directory (AD) SSO Integration

Follow these steps to set up Azure AD as an identity provider (IdP) for secure Single Sign-On (SSO).

Azure Portal → App Registrations → Authentication → API Permissions

  1. Access Azure Active Directory
    • Log in to the Microsoft Azure Portal.
    • Go to Azure Active Directory > App Registrations.
    • Click New Registration.
  2. Register a New Application
    • Enter a Name (e.g., "MyApp SSO").
    • Select Accounts in this organizational directory only (for single-tenant).
    • Under Redirect URI, select Web and enter:
      ⚠️ Required Callback URL:
      https://yourdomain.com/sso/azure/callback
      This must be exactly /sso/azure/callback for authentication to work.
    • Click Register.
  3. Configure Authentication Settings
    • Go to Authentication.
    • Ensure ID Tokens is checked.
    • Add additional redirect URIs if needed (e.g., for dev/staging environments).
  4. Set API Permissions
    • Navigate to API Permissions.
    • Click Add a permission > Microsoft Graph.
    • Add delegated permissions:
      • openid
      • profile
      • email
    • Click Grant admin consent.
  5. Generate Client Secret
    • Go to Certificates & Secrets.
    • Click New client secret.
    • Set an expiration (recommended: 12 months).
    • Copy the Secret Value (you won’t see it again).
  6. Configure Your Application
    • In your app’s SSO settings, enter:
      • Domain
      • Azure Tenant ID (found in Azure AD > Overview)
      • Application (Client) ID
      • Client Secret
      • Callback URL: /sso/azure/callback
🔍 Troubleshooting
  • Error: "AADSTS50011 - Redirect URI mismatch" → Verify the callback URL is registered in Azure AD.
  • Error: "Invalid client secret" → Generate a new secret in Azure.
  • Error: "Insufficient permissions" → Re-grant admin consent in API Permissions.

Local Authentication (No SSO)

Gheetah's built-in authentication system provides a secure local user management solution without requiring external identity providers. This is the default option during initial setup.

🚀 Zero Configuration Required
No additional setup is needed. Simply select "No SSO" during initial configuration to activate Gheetah's native authentication system.
Key Features:
  • Automatic JWT-Based Authentication
    Gheetah automatically implements OpenID-compatible authentication using JWT Bearer tokens for secure API access.
  • Seamless User Onboarding
    Custom registration and login endpoints are generated automatically when you choose this option.
  • Complete User Management
    • User self-registration (configurable)
    • Secure password reset functionality
    • Account locking after failed attempts
    • Email verification workflows
Implementation Details:

When you select "No SSO" during initial setup:

  1. Gheetah generates:
    • account/register endpoint (POST)
    • account/login endpoint (POST)
    • JWT token generation/validation system
  2. All authentication flows use:
    • OpenID-compatible JWT tokens
    • HS256 signing algorithm (configurable)
    • Secure HTTP-only cookies for web sessions

Users created through this system are immediately available in Gheetah with default basic user permissions (configurable in admin panel).

Agent Operation Workflow

Initial Agent Setup Process

After successfully installing via GheetahAgentSetup.exe and launching the Agent, you'll be presented with a CLI interface that clearly guides you through the registration process.

Registration Procedure

1
Provide Gheetah URL

The Agent will prompt you to enter your Gheetah Test Orchestration instance's root URL (without a trailing "/").

Example: https://your-gheetah-instance.com (correct) vs https://your-gheetah-instance.com/ (incorrect)
2
Registration Request

Upon submitting the URL, the Agent sends a registration request to your Gheetah server.

3
Admin Approval

The request appears as "Pending Request" in the Gheetah admin interface. Once approved by an administrator, your Agent becomes officially registered and ready for remote test execution.

Test Execution Workflow

1. Test Initiation

When you select your registered Agent from the Agent list and initiate test execution for specific scenarios, Gheetah:

  • Packages the relevant project into a .zip archive
  • Transmits it to the Agent's environment
2. Project Deployment

The Agent:

  • Receives the .zip package in its temp directory
  • Extracts the project contents
  • Navigates into the BDD project structure
3. Test Execution

The Agent executes the specified scenarios while:

  • Monitoring the execution in real-time
  • Streaming runtime results back to the Gheetah server
4. Results & Cleanup

After execution completes:

  • Results are available in Gheetah as both HTML reports and CLI output
  • The Agent automatically cleans up by removing the temporary .zip and project files
  • System resources are freed for subsequent executions
Architecture Benefits

This distributed execution model provides:

  • Resource efficiency: No persistent storage usage on agent machines
  • Isolation: Clean environment for each test run
  • Scalability: Multiple agents can execute tests in parallel
  • Security: Sensitive data isn't retained on agent machines

Pending Request Security Mechanism

Security-First Approach

The Pending Request system serves as a critical security gateway, ensuring only authorized Agents can register with your Gheetah Test Orchestration instance and perform remote test executions.
